📐 Frame-to-Blender Aligner

Refines imprecise GUI traces with CV, then builds a pixel-aligned 3D Blender scene

Problem it solves

Hand-traced GUI Frame Tool JSON can be off by tens of pixels. Observed on Band Ritual: the ON/OFF button trace was ~90 px above the actual PNG position. Running that imprecise JSON directly into Blender produces 3D geometry that visibly does not line up with the underlying artwork, and you spend iterations nudging elements by hand in Blender.

This agent closes the gap with a CV refinement pass plus a render-based validator, so every element ends up where the PNG says it is — automatically.

Pipeline

shapes.json (rough trace)                  reference.png
          |                                       |
          v                                       |
   [refine_shapes.py]  <---------- same PNG ------+
          |
          v
   refined.json  -----> [build_blender.py] -----> Blender scene
                                  |
                                  v
                          render.png --> [validate.py] --> alignment_report.json

Files

  • refine_shapes.py — Hough circles + Canny-edge snap refinement for circles / rects / rings. Bounded search radius so the agent cannot invent new elements, only tighten positions.
  • build_blender.py — Blender scene builder. Biggest rect = chassis frame. Shadeless Emission chassis (PNG displays as-is), Principled BSDF knobs with bevel + subdiv + dome + indicator. UV-crops the PNG to the actual frame area so shapes and artwork align pixel-perfect.
  • validate.py — Renders top-down ortho, per-shape pixel diff vs reference PNG, flags any element with alignment score < 0.6. Report is JSON-serializable so the next iteration can re-refine only flagged items.
  • README.md — Full context, design decisions, usage.

One-liner

python3 refine_shapes.py shapes.json reference.png refined.json \
  && blender --background --python build_blender.py -- refined.json reference.png \
  && python3 validate.py refined.json reference.png render.png report.json

Design decisions

  • No element semantics. Every circle becomes a 4-piece knob, every rect becomes a recessed panel. The agent never decides "this circle is a knob and that is a toggle button" — semantic reclassification is a separate pass (see sibling image-to-blender).
  • Frame-relative coordinates. Biggest rect in JSON = chassis boundary. Everything else is (px - FRAME_X) / FRAME_W, so the same JSON works at any chassis render size.
  • CV refines, never invents. Refinement is bounded (±30% radius, ±15% rect side). If CV finds nothing convincing the traced value is kept.
  • Shadeless chassis. PNG texture uses an Emission shader so the artwork renders exactly as it looks in the file — no environment tinting or specular shift.

Slot in the pipeline

Moodboard → VST GUI Pipeline (ChatGPT PNG) → GUI Frame Tool (trace) → Frame-to-Blender Aligner (you are here) → Blender Redesign (final polish + render)

Sibling agent Image-to-Blender goes the other way (auto-detection, no manual trace). Use Aligner when the autodetector misses decorations or dropdown slots the user traced by hand.
