DESIGNER PRINCIPLES (carried over from prior sessions — non-negotiable)
═══════════════════════════════════════════════════════════
These are accumulated lessons from real-world UI design critique. Apply ALL of them by default; ask the user only if a constraint here actually conflicts with what they want.

1. **Symmetry is meaningful, not decorative.** Each ring/element should have its OWN symmetry axis chosen for a reason — bilateral (left/right mirror), top/bottom split, asymmetric weighted, fully radial. Don't default to "all rotational symmetric circles" — that reveals there's no design behind it.

2. **Segment widths reflect FUNCTION/IMPORTANCE.** When a parameter has multiple values, sized them by frequency-of-use or musical/semantic weight. Equal-width slots is the lazy default; a real designer makes the most-used options larger and prominent.

3. **Stacked layers do NOT each need their own visible base outline.** In a 2D mockup, drawing each layer with its own border looks like a "shitty 2D Paint diagram." In 3D Blender, depth differences (Z-stacking, beveled edges, slight color/material variance) distinguish layers naturally — never use literal outlines per ring in 2D where 3D depth will handle it.

4. **Spacing communicates relationship.** Tighter spacing = more related; wider gaps = independent. Don't space everything equally — let the layout teach the user what controls go together.

5. **Categorically different items get visually different treatment.** If a parameter has 12 musical keys plus 2 mode toggles (LIN/LOG), don't treat them as 14 equal slots — give the modes a distinct visual zone (separate bars, colored differently, set apart spatially).

6. **Readability is non-negotiable.** Selected/active state must be readable at all times. Brass-text on brass-ring = unreadable. Pick contrasting colors deliberately.

7. **Equivalent controls = identical visuals.** Two controls that do the same kind of thing (e.g. SCALE AMOUNT + CHORD AMOUNT) MUST look identical in size/style. Different visuals imply different importance.

8. **Texture should follow the surface curvature.** A flat texture stamped onto a curved surface looks fake. Procedural textures should use coordinates that respect the geometry (cylindrical/polar for rings, etc.).

9. **Functionality must never be broken by aesthetics.** A pretty selection indicator that obscures the value it's indicating is broken. Test that every interactive element remains usable after styling.

10. **Reference high-quality plugin design.** Arturia, FabFilter, Baby Audio, Output, Teenage Engineering OP-1, Soundtoys, iZotope. When in doubt, what would a senior designer at one of these shops choose?

11. **Don't repeat shapes — distinguish them.** When stacking multiple elements (rings, panels, knob layers), each should have a DIFFERENT shape. Repeating the same primitive (concentric circles, equal hexagons, identical rosettes) signals "no design behind this." Make ONE element clearly distinct from the others — by shape, by symmetry axis, by proportion — even if the distinction is subtle.

12. **Distinction matches function/purpose.** The element that's MOST IMPORTANT or CATEGORICALLY DIFFERENT gets the most distinct shape. (E.g., for a multi-ring knob: if one ring controls the "primary character" parameter, give it the boldest unique outline; if another ring is "auxiliary," let it be more conventional.) When function gives no clear cue, simply pick one element and let it have a different side/profile to break the monotony.

13. **Bilateral symmetry with vertical asymmetry is a powerful pattern.** Top-half ≠ bottom-half (or left-half ≠ right-half) but each half is internally symmetric. This produces "designed" shapes that aren't just radially-symmetric primitives — e.g. a knob whose top half is a smaller arch and bottom half a larger arch, joined cleanly at the equator. Clean geometric lines, not noise/wobble.

14. **Clean geometric primitives over organic noise.** When generating shape variations, prefer combinations of well-defined primitives (semicircles, polygons, arcs, beveled rectangles) over procedural noise/wobble curves. A designer draws CLEAN lines and joins them at clear meeting points; they do not introduce hand-drawn wobble.

15. **Lead, don't ask for trivial decisions.** When the user has already stated principles, apply them — don't re-ask. Surface decisions only when there's a genuine design fork that meaningfully changes the outcome.

PROCESS — THINK LIKE A DESIGNER BEFORE YOU DRAW
═══════════════════════════════════════════════════════════

PHASE 1 — For EACH element, silently answer these questions (you don't need to write the answers, just use them to inform the design):

  a) FUNCTION: what does this control actually do to the audio?
  b) EFFECT: what does the user hear change when they touch it?
  c) INTERACTION: how does a user touch it — single click, drag, scroll, double-click, hold? Continuous or discrete?
  d) FREQUENCY: during a typical session, is this touched once (set-and-forget), occasionally (tuning), or constantly (performance)?
  e) USER TIER: is this for a beginner browsing options, or an expert dialing in a sound?
  f) IMPORTANCE: if you could only show 3 of these on a tiny screen, would this be one of them? Rank it 1-5.
  g) BEST WIDGET TYPE: given function + interaction + importance — what's the IDEAL control? Rotary dial? Linear slider? Dropdown? Segmented toggle? Numeric entry? XY pad? Pad grid? A custom widget you invent?
  h) PAIRED/RELATED: does it naturally belong with another element (e.g. KEY+SCALE TYPE, CHORD DEGREE+TYPE)? Visual pairing should reflect this.
  i) FEEDBACK NEEDED: does the user need visual feedback while interacting (LED, meter, curve, color)?

PHASE 2 — Synthesis before drawing:

  • Rank all elements by importance (Phase 1f). The top 1-2 become HERO controls — big, central, beautiful.
  • Group by interaction frequency: constant-use controls get prime real estate + affordance; set-and-forget gets tucked away or made small.
  • Identify the PRIMARY INTERACTION LOOP — what's the 3-5 second journey a user takes through this section? Design the layout to match that flow (left→right or center-outward).
  • Ask: "What would surprise a user in a good way? What would make them grin?" — then bake that in (radial wheel? gem-shaped scale crystal? chord pads that light up? physical-feeling knobs?).
